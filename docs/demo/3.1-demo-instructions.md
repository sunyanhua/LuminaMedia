## 1. 演示环境说明

### 1.1 访问信息
- **前端应用**: http://localhost:5174/government
- **后端API**: http://localhost:3003
- **API文档**: http://localhost:3003/api/docs (Swagger UI)
- **健康检查**: http://localhost:3003/health
- **监控面板**: http://localhost:8080 (SkyWalking APM)

### 1.2 系统要求
- **操作系统**: Windows 10/11, macOS 10.15+, Linux Ubuntu 20.04+
- **内存**: 至少8GB RAM (推荐16GB)
- **存储**: 至少10GB可用空间
- **网络**: 可访问互联网（用于AI API调用）
- **Docker**: Docker Desktop 4.12+ 或 Docker Engine 20.10+

### 1.3 快速启动命令
```bash
# 克隆项目（如未克隆）
git clone https://github.com/your-org/LuminaMedia.git
cd LuminaMedia

# 启动所有服务
docker-compose up -d

# 查看服务状态
docker-compose ps

# 查看日志
docker-compose logs -f app

# 停止服务
docker-compose down
```

### 1.4 数据库初始化
DEMO数据已通过以下脚本自动初始化：
1. `scripts/17-demo-government-users-init.sql` - 政务版DEMO用户和角色
2. `scripts/23-demo-tenant-profile.sql` - 租户单位画像数据
3. `scripts/24-demo-reference-infos.sql` - 参考信息数据
4. `scripts/21-sentiment-table.sql` - 舆情监测数据

如需重新初始化数据：
```bash
# 进入数据库容器
docker-compose exec db-lumina mysql -u lumina_user -plumina_password lumina_media

# 或在主机执行
cat scripts/17-demo-government-users-init.sql | docker-compose exec -T db-lumina mysql -u lumina_user -plumina_password lumina_media
```

## 4. 常见问题解答 (FAQ)

### 4.1 环境部署问题
**Q: Docker启动失败，端口被占用怎么办？**
A: 修改 `docker-compose.yml` 中的端口映射，如将 `"3003:3003"` 改为 `"3004:3003"`，前端端口同理。

**Q: AI API密钥无效或配额不足怎么办？**
A: 使用备用API密钥，或在 `.env` 文件中配置多个密钥（逗号分隔），系统支持自动轮询。

**Q: 数据库连接失败怎么办？**
A: 检查MySQL容器是否正常启动：`docker-compose logs db-lumina`，确保初始化脚本执行成功。

**Q: 前端访问显示空白页怎么办？**
A: 检查前端容器日志：`docker-compose logs dashboard`，可能是构建失败或依赖问题。

### 4.2 功能使用问题
**Q: 登录时提示"用户名或密码错误"**
A: 确保使用正确的DEMO账号和密码 `LuminaDemo2026`，检查数据库用户数据是否初始化。

**Q: AI生成内容速度慢**
A: 可能是网络延迟或API限流，可以稍等重试，或切换到本地Qwen模型（如已配置）。

**Q: 三审三校流程卡住**
A: 检查用户角色权限，确保当前用户有审批权限，或检查审批状态是否已推进。

**Q: 一键报告生成失败**
A: 检查AI服务配置，报告生成依赖Gemini API，确保API密钥有效且网络可访问。

**Q: Word导出功能报错**
A: 当前Word导出为模拟实现，如报错可忽略，演示时说明此为模拟功能，正式版会完善。

### 4.3 演示过程问题
**Q: 演示时间紧张，哪些环节可以压缩？**
A: 可压缩的环节：智慧档案（重点展示单位画像）、参考信息（快速演示一键采用）、运营监测（重点展示一键报告）。

**Q: 观众对某个功能特别感兴趣，想多看怎么办？**
A: 准备深度演示路径：如对AI生成感兴趣，可展示不同内容类型的生成；如对审批流程感兴趣，可展示驳回和修改流程。

**Q: 网络突然中断怎么办？**
A: 切换到离线演示模式：使用预录的屏幕录像、功能截图，结合讲解继续演示。

**Q: 观众提出超出DEMO范围的需求怎么办？**
A: 记录需求，表示后续版本会考虑，引导到现有功能的优势，或安排专项交流。

### 4.4 技术细节问题
**Q: 平台采用什么技术架构？**
A: 后端NestJS + TypeORM + MySQL，前端React + Vite + Ant Design Mobile，容器化部署，模块化单体架构。

**Q: 数据安全如何保障？**
A: 政务版支持私有化部署，数据完全独立；多重加密、访问控制、操作审计；符合政府信息系统安全要求。

**Q: 是否支持与其他系统集成？**
A: 提供标准RESTful API，支持与OA系统、门户网站、数据中台等集成；支持Webhook和消息队列。

**Q: 平台的扩展性如何？**
A: 采用模块化设计，新功能可独立开发部署；支持插件机制；云原生架构支持水平扩展。

## 6. 应急处理方案

### 6.1 服务故障
**症状**: 页面无法访问、API返回错误
**处理步骤**:
1. 快速检查服务状态: `docker-compose ps`
2. 查看错误日志: `docker-compose logs [服务名]`
3. 尝试重启服务: `docker-compose restart [服务名]`
4. 如无法快速修复，切换到备用演示方案（录屏+讲解）

### 6.2 AI服务异常
**症状**: AI生成失败或超时
**处理步骤**:
1. 检查网络连接
2. 切换到备用API密钥
3. 如仍失败，使用预生成的内容演示
4. 解释为网络临时问题，不影响产品能力

### 6.3 数据库问题
**症状**: 数据加载失败、用户登录失败
**处理步骤**:
1. 检查数据库容器状态
2. 重新执行初始化脚本
3. 如时间紧张，使用预登录的浏览器会话
4. 解释为本地环境问题，正式环境有高可用保障

### 6.4 浏览器兼容性问题
**症状**: 页面样式错乱、功能异常
**处理步骤**:
1. 切换到Chrome浏览器
2. 清除浏览器缓存
3. 如仍有问题，使用备用设备
4. 说明正式环境经过多浏览器测试

### 6.5 演示时间不足
**处理步骤**:
1. 优先保证核心流程演示（登录→创作→审批→发布）
2. 压缩非核心环节讲解时间
3. 提供详细资料供后续查阅
4. 安排二次专项演示
