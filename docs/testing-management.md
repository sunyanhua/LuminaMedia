# LuminaMedia 测试文件管理规范

## 1. 概述

为确保项目代码文件的整洁和易管理，制定本测试文件管理规范。所有测试文件必须统一存放于 `test/` 目录下，按照与 `src/` 目录相同的结构组织，禁止在源代码目录中随意创建测试文件。

## 2. 目录结构规范

### 2.1 测试文件存放位置
- **单元测试文件**: `test/` 目录下，结构与 `src/` 目录一一对应
- **集成测试文件**: `test/integration/` 目录下
- **端到端测试文件**: `test/e2e/` 目录下（已有）
- **测试工具和辅助文件**: `test/helpers/` 目录下

### 2.2 目录映射规则
```
项目根目录/
├── src/
│   ├── modules/
│   │   ├── auth/
│   │   │   ├── controllers/
│   │   │   ├── services/
│   │   │   └── ...
│   │   └── ...
│   └── shared/
│       └── ...
└── test/
    ├── modules/
    │   ├── auth/
    │   │   ├── controllers/
    │   │   ├── services/
    │   │   └── ...
    │   └── ...
    ├── shared/
    │   └── ...
    ├── integration/
    └── e2e/
```

### 2.3 文件命名规范
- 单元测试文件: `[被测试文件名].spec.ts`
- 集成测试文件: `[被测试模块].integration.spec.ts`
- 端到端测试文件: `[功能模块].e2e-spec.ts`
- 测试辅助文件: `[功能描述].helper.ts` 或 `[功能描述].fixture.ts`

## 3. 迁移计划

### 3.1 迁移范围
所有现有 `.spec.ts` 和 `.test.ts` 文件从 `src/` 目录迁移到 `test/` 目录。

### 3.2 迁移步骤
1. **创建目标目录结构**: 在 `test/` 下创建与 `src/` 对应的目录结构
2. **文件迁移**: 将测试文件从 `src/` 复制到 `test/` 对应位置
3. **更新导入路径**: 修改测试文件中的相对导入路径，指向 `src/` 目录下的源代码
4. **验证测试**: 运行迁移后的测试文件，确保全部通过
5. **清理源文件**: 确认测试通过后，删除 `src/` 目录中的原始测试文件

### 3.3 导入路径更新规则
迁移前（在 `src/` 目录中）:
```typescript
import { AuthService } from '../services/auth.service';
import { User } from '../../entities/user.entity';
```

迁移后（在 `test/` 目录中）:
```typescript
import { AuthService } from '../../../src/modules/auth/services/auth.service';
import { User } from '../../../src/entities/user.entity';
```

**路径计算规则**: 从 `test/` 中的测试文件到 `src/` 中的源代码文件，使用相对路径 `../../../src/` 作为基准。

## 4. Jest配置调整

### 4.1 根目录配置
在 `jest.config.js` 中配置 `roots` 和 `testMatch`:
```javascript
module.exports = {
  moduleFileExtensions: ['js', 'json', 'ts'],
  rootDir: '.',
  testRegex: '.*\\.spec\\.ts$',
  testPathIgnorePatterns: ['/node_modules/', '/dist/'],
  transform: {
    '^.+\\.(t|j)s$': 'ts-jest',
  },
  collectCoverageFrom: [
    'src/**/*.(t|j)s',
    '!src/**/*.spec.ts',          // 排除src中的测试文件（迁移后应无）
    '!test/**/*.ts',              // 排除test目录中的所有文件
  ],
  coverageDirectory: './coverage',
  testEnvironment: 'node',
};
```

### 4.2 模块路径映射
在 `tsconfig.json` 中配置路径别名，简化测试文件导入:
```json
{
  "compilerOptions": {
    "baseUrl": "./",
    "paths": {
      "@src/*": ["src/*"],
      "@test/*": ["test/*"]
    }
  }
}
```

使用路径别名导入:
```typescript
import { AuthService } from '@src/modules/auth/services/auth.service';
```

## 5. 测试文件创建和管理规则

### 5.1 新建测试文件流程
1. **确定位置**: 根据被测试文件在 `src/` 中的位置，确定 `test/` 中的对应目录
2. **创建目录**: 如目录不存在，先创建对应目录结构
3. **创建文件**: 在正确位置创建测试文件，遵循命名规范
4. **编写测试**: 使用标准测试结构（describe/it）
5. **验证路径**: 确保导入路径正确指向 `src/` 目录

### 5.2 测试文件删除规则
1. **源代码删除**: 当 `src/` 中的源代码文件被删除时，对应的测试文件必须同步删除
2. **测试重构**: 当测试逻辑过时或需要重构时，直接修改现有测试文件
3. **临时测试**: 临时测试文件必须添加 `// TODO: 临时测试，需在 [日期] 前删除` 注释

### 5.3 测试文件审查
1. **代码审查**: 所有测试文件变更必须经过代码审查
2. **覆盖率要求**: 新增功能必须包含测试，核心功能测试覆盖率要求100%
3. **命名规范审查**: 审查测试文件命名是否符合规范

## 6. 违规处理

### 6.1 禁止行为
1. ❌ 在 `src/` 目录中创建新的测试文件
2. ❌ 在根目录或其他非 `test/` 目录创建测试文件
3. ❌ 使用非标准命名（如 `.test.ts` 代替 `.spec.ts`）
4. ❌ 测试文件导入路径指向 `test/` 目录外的其他测试文件

### 6.2 违规检测
1. **预提交钩子**: 配置Git预提交钩子，检查新测试文件位置
2. **CI/CD检查**: 在CI/CD流水线中添加测试文件位置检查
3. **定期审计**: 每月执行一次测试文件位置审计

## 8. 备份管理策略

### 8.1 备份目的
- **安全迁移**: 确保迁移过程中原始测试文件不丢失
- **回滚能力**: 如迁移后发现问题，可快速恢复原始状态
- **审计追踪**: 保留历史版本，便于对比和审计

### 8.2 备份位置和结构
```
项目根目录/
├── backup_test_files/           # 迁移前测试文件备份目录
│   ├── [原始文件名].spec.ts    # 备份文件，保持原始内容
│   └── README.md               # 备份目录说明
└── test/                       # 迁移后的测试文件目录
```

### 8.3 备份保留策略
1. **保留期限**: 备份文件保留30天，确保迁移稳定后清理
2. **清理条件**:
   - 所有迁移后的测试文件正常运行30天
   - 无回滚需求
   - 项目进入下一阶段开发
3. **清理命令**:
   ```bash
   # 清理备份目录（需谨慎操作）
   rm -rf backup_test_files/
   ```

### 8.4 备份验证
- **完整性检查**: 确保备份文件与原始文件内容一致
- **可恢复性验证**: 从备份恢复测试文件，确保能正常运行
- **目录结构验证**: 备份目录结构清晰，便于查找

## 9. 附录

### 9.1 常见问题解答

**Q: 为什么要把测试文件移动到单独的 `test/` 目录？**
A: 1) 保持 `src/` 目录纯净，只包含生产代码；2) 提高代码组织结构清晰度；3) 便于测试文件统一管理；4) 符合许多大型项目的实践。

**Q: 迁移后测试运行会变慢吗？**
A: 不会。Jest测试运行速度主要取决于测试数量和复杂度，而不是文件位置。合理的目录结构可能还会提高模块缓存效率。

**Q: 如何处理测试中使用的测试辅助函数？**
A: 将测试辅助函数放在 `test/helpers/` 目录下，通过 `import { helper } from '../../helpers/test-helper';` 导入。

**Q: 迁移过程中测试失败怎么办？**
A: 1) 检查导入路径是否正确；2) 确保模块导出没有改变；3) 验证测试逻辑仍然有效；4) 回滚更改，分析问题原因。
