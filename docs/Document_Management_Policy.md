# LuminaMedia 项目文档管理制度

**版本**: v1.0  
**生效日期**: 2026-04-05  
**适用范围**: LuminaMedia项目全生命周期

---

## 1. 总则

### 1.1 目的
为规范LuminaMedia项目文档的管理，确保文档的完整性、准确性和可追溯性，特制定本制度。

### 1.2 管理原则
- **统一规范**: 所有文档遵循统一的命名、存储和版本规范
- **分级管理**: 按重要性和时效性对文档进行分级管理
- **及时归档**: 临时文档和过时文档及时清理归档
- **版本追溯**: 重要文档保留版本历史，确保可追溯

## 3. 文档命名规范

### 3.1 通用命名规则
- **英文文档**: 使用PascalCase，如 `Project_Completion_Report.md`
- **中文文档**: 使用中文描述，如 `功能菜单列表.md`（仅限必要场景）
- **版本号**: 文档头部标注版本，如 `**版本**: v1.0`
- **日期格式**: YYYY-MM-DD，如 `2026-04-05`

### 3.2 任务清单命名
```
phase-{N}-{description}.md
```
示例：
- `phase-1-database-migration.md`
- `phase-2-backend-services.md`

### 3.3 报告类文档命名
```
{Type}_{Date}_{Description}.md
```
示例：
- `Audit_20260404_全量质检第4轮.md`
- `Deployment_Acceptance_Report.md`

## 4. 文档版本管理

### 4.1 版本号规则
采用 `主版本.次版本` 格式：
- **v1.0**: 初始版本
- **v1.1**: 小幅更新
- **v2.0**: 重大更新

### 4.2 版本信息格式
文档头部必须包含：
```markdown
**版本**: v1.0
**创建日期**: 2026-04-05
**更新日期**: 2026-04-05
**作者**: LuminaMedia Team
**状态**: 草稿/审核中/已发布/已归档
```

### 4.3 变更记录
重要文档更新时，需在文档末尾添加变更记录：
```markdown
## 变更记录
| 版本 | 日期 | 变更内容 | 作者 |
|------|------|----------|------|
| v1.1 | 2026-04-05 | 更新任务清单路径 | xxx |
| v1.0 | 2026-04-04 | 初始版本 | xxx |
```

## 5. 文档生命周期管理

### 5.1 文档状态
- **草稿**: 正在编写中，未完成
- **审核中**: 已完成编写，等待审核
- **已发布**: 审核通过，正式使用
- **已过时**: 内容已过时，有替代文档
- **已归档**: 已移至archive目录，仅供参考

### 5.2 状态转换流程
```
草稿 → 审核中 → 已发布 → 已过时 → 已归档
```

### 5.3 归档条件
符合以下条件的文档应及时归档：
1. 版本已过时，有新版本替代
2. 项目阶段已完成，文档不再更新
3. 临时性文档，使用期限已过
4. 内容已被其他文档合并或取代

### 5.4 归档操作
1. 在文档头部添加归档标记：
   ```markdown
   ⚠️ **此文档已归档**: 内容已过时，仅供参考
   **归档日期**: 2026-04-05
   **替代文档**: [新文档链接](./new-doc.md)
   ```
2. 将文档移至 `docs/archive/deprecated/` 或 `tasks/archive/{version}/`
3. 更新文档索引（README.md）中的链接状态

## 6. 临时文档管理

### 6.1 临时文档定义
- 仅供短期使用的参考文档
- 会议记录、讨论纪要等
- 测试数据、实验性方案

### 6.2 临时文档存储
- 临时文档统一存放在 `/tmp/docs/` 或 `/tmp/tasks/`（需创建）
- 或在文件名中标注 `temp-` 前缀

### 6.3 临时文档清理
- 临时文档使用期限不超过30天
- 到期后如无保留价值，直接删除
- 如有参考价值，归档保存

## 7. 文档链接管理

### 7.1 内部链接规范
- 使用相对路径，如 `./docs/README.md`
- 路径变更时，必须更新所有相关链接
- 定期检查死链

### 7.2 链接更新流程
当文档移动或重命名时：
1. 使用全文搜索查找所有引用位置
2. 逐一更新链接
3. 验证链接有效性

## 9. 质量控制

### 9.1 审核要求
- 重要文档（实施方案、设计文档）必须经过审核
- 审核人应在文档中标注审核意见
- 审核不通过的文档返回修改

### 9.2 定期检查
- 每月检查一次文档链接有效性
- 每季度评估一次文档时效性
- 每版本发布时全面更新文档状态

### 9.3 文档准确性
- 文档内容必须与代码实现保持一致
- API文档必须与接口实现同步更新
- 配置文档必须与实际配置一致
